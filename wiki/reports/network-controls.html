1 · Defense-in-depth perimeter (outer → inner)

The outermost ring is the optional Cloudflare front-door ROADMAP — §4 recipe; everything from the in-stack WAF inward is measured / gate-proven. A request crosses every ring in order; the in-stack WAF is the sole public listener (Kong sits behind it).

Cloudflare front-door (ROADMAP — hosted only)
DNS-proxy (orange-cloud) · managed WAF + OWASP ruleset · rate-limit 600/60s · Turnstile bot-challenge · Full-strict TLS + authenticated origin pull (mTLS).
In-stack OWASP-CRS WAF (sole public listener)
nginx + ModSecurity v3 + CRS v4 (owasp/modsecurity-crs:4-nginx). SecRuleEngine On, paranoia 2, inbound anomaly threshold 5. Only container binding a public port.
Kong edge gateway (behind the WAF)
Bound to 127.0.0.1 for dev, never 0.0.0.0. ip-restriction on /admin/v1/* + tenant-control + adapter-registry (private CIDRs only).
net-edge plane (public-facing services)
waf, kong, studio, playground, gotrue, postgrest, realtime, grafana — WAN ingress.
net-control plane (internal: true — no WAN egress)
tenant-control, orchestrator, permission-engine, schema-service, webhook-dispatcher, function-scheduler, vault. Reachable only via the dual-attached routers.
net-data plane (internal: true — no WAN egress)
postgres, mysql, mariadb, cockroach, mssql, mongo, redis, minio. Owner-scope/RLS ABAC enforced per request at the front-door routers — no raw socket from the edge.

The additive overlay docker-compose.netseg.yml splits the stack into four default-deny bridges; two containers reach each other iff they share a bridge. 5 · Cloudflare front-door recipe (ROADMAP — hosted/managed deploy)

For a hosted Grobase deploy, put Cloudflare in front of the in-stack WAF as a second, defense-in-depth perimeter (the in-stack WAF stays on — belt and braces). These are a deployment recipe — ROADMAP, not a measured in-stack control. The real config knobs from §4 of the doc:

1 · DNS — proxied (orange-cloud) step 1
cloudflare_record: type=A, value=origin_ip (host running the in-stack WAF :443), proxied=true (ORANGE cloud), ttl=1. Origin :443 firewalled to Cloudflare IP ranges only.
2 · WAF managed + custom rules step 2
cloudflare_ruleset phase=http_request_firewall_managed: execute Cloudflare Managed Ruleset (id efb7b8c9…) + OWASP Core Ruleset (id 4814384a…). Custom rule blocks /admin/v1/ at the edge too (defense in depth).
3 · Rate-limiting step 3
phase=http_ratelimit on /data/v1/: characteristics [cf.colo.id, ip.src], period 60s, 600 req/period, mitigation_timeout 600. Complements the in-stack per-tenant token bucket (m51).
4 · Bot management / Turnstile step 4
managed_challenge on /auth/v1/signup or cf.client.bot_score < 30. Pair with a Turnstile widget; verify cf-turnstile-response server-side before issuing a session.
5 · Full-strict TLS + AOP (mTLS to origin) step 5
ssl="strict" (Full strict), min_tls 1.2, tls_1_3 on, always_use_https. authenticated_origin_pulls enabled; on the WAF nginx: ssl_client_certificate cloudflare-origin-pull-ca.pem + ssl_verify_client on → cryptographic origin lock.
6 · Cache rules step 6
phase=http_request_cache_settings: NEVER cache /data/v1/ or /auth/v1/ (per-tenant + authenticated); cache static assets (js/css/png/svg/woff2) at edge_ttl 86400.

6 · Honest framing — the one parity note

Where Grobase does not outright win in-stack, the path is explicit and labelled.

Self-host perimeter — Grobase WINS WIN
For a self-hosted deploy Grobase ships an OWASP-CRS WAF as the sole public listener and per-plane segmentation that Supabase OSS simply does not have. Plus full data residency: all of the above runs entirely in the operator's own infra.
Volumetric DDoS scrubbing — parity-via-Cloudflare parity
In-stack we have a per-tenant token bucket (m51) + WAF CRS. Edge volumetric DDoS scrubbing needs the §4 Cloudflare front-door. Supabase managed leans on its own Cloudflare; the §4 recipe gives a Grobase hosted deploy the same outer perimeter — on top of the in-stack WAF.
Managed IP-restriction dashboard — parity-via-own-VPC parity
Supabase's network restrictions are a polished managed dashboard (paid). Grobase's per-tenant allowlist is config + a control-plane endpoint, enforced on every API call and re-verifiable by the operator (m106). A dashboard toggle is a DX nicety, not a control gap.
Origin mTLS lock (AOP) — parity once §4.5 applied parity
Authenticated Origin Pulls is a deployment step (nginx ssl_verify_client on, mount Cloudflare's origin-pull CA). Once applied, a request that did not come through Cloudflare is rejected at TLS — parity with Supabase managed AOP.
