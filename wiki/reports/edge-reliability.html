What the suite found — and we fixed

The 1381-vector sweep is only worth running if it changes the code. It did. Three concrete defects were surfaced and closed; one "leak" was proven a false positive.

Oversize body → was 500, now 413 fixed
A >limit request body crashed into a generic 500. AllExceptionsFilter now honors the carried 4xx status — a 2 MB body returns 413 Payload Too Large, verified.
Malformed op body → was 500, now 400 fixed
A non-string op (e.g. an array) hit a TypeError in dto.resolveOp → 500. query.service now guards the op shape — a malformed array op returns 400, verified.
'/etc/passwd' leak → FALSE POSITIVE no leak
The server returned a clean 400 echoing the rejected identifier. The old assertion matched the path string, not file content. It now matches passwd CONTENT ('root:…:0:0:'), not the path. No real leak ever existed.

Methodology

Honest, reproducible, and deliberately paced.

1381 distinct vectors, one per corpus row, each rewriting path / method / body / auth headers from postman/corpus/edge-corpus.json.

Invariant model. Every vector asserts the same three things: (1) never a 5xx on a real op, (2) a valid HTTP status in the declared set, (3) no content leak (no stack traces, no source paths, no file contents in the body).

Pacing. Run with --delay-request 60 and --timeout-request 15000 so the suite stresses the contract, not the socket. The remaining tail is the data-plane 5 s forward timeout + newman 15 s socket timeout colliding under a 1381-request burst on a shared 24,888-tenant box — environmental, and reproducible-clean on a quiet/dedicated control plane.
