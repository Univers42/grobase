# data-plane-pool

The concrete engine adapters and connection pooling for the Rust data plane: it implements `data-plane-core`'s `EngineAdapter` / `EnginePool` / `PoolRegistry` traits for eight database engines, holds long-lived mount-keyed connection pools, resolves credentials, and enforces multi-tenant isolation on every request.

## Role in the workspace

The `data-plane-router` cargo workspace is the Rust data plane of the mini-baas / Grobase BaaS. It has four crates:

| Crate | Responsibility |
|-------|----------------|
| `data-plane-core` | Pure domain vocabulary + traits (`EngineAdapter`, `EnginePool`, `PoolRegistry`, `DatabaseMount`, `DataOperation`, `EngineCapabilities`, `Isolation`, `RequestIdentity`). No I/O. |
| **`data-plane-pool`** (this crate) | The concrete I/O: engine adapters + pooling + credential resolution + per-request isolation. |
| `data-plane-server` | The axum HTTP server. |
| `engine-conformance` | Cross-engine test battery. |

This crate is the only place in the workspace that talks to a real database. It implements core's traits and is consumed by `data-plane-server`:

- Each engine ships an `EngineAdapter` (e.g. `PostgresEngineAdapter`, `MysqlEngineAdapter`, `MongoEngineAdapter`, …). The server builds the live adapter list and hands it to a `DefaultPoolRegistry`.
- `DefaultPoolRegistry` implements core's `PoolRegistry` — `get_or_create(mount)` returns a `Box<dyn EnginePool>`; the server then calls `execute(operation, identity)` / `begin(request)` on that pool.
- The server owns config and wires the credential pipeline (`EnvMountResolver::from_config`, `ProviderRegistry`), the shared-pool flag (`DefaultPoolRegistry::with_config`), rotation (`AppState::rotate` over `POST /v1/admin/rotate`), and the capability descriptors served at `/v1/capabilities`.

It replaces the legacy TypeScript query-router behaviour of constructing a fresh `new Client()` / `new MongoClient()` per request with long-lived, mount-keyed pools.

## Mental model

**The resolve → register → credential → pool → execute pipeline.** A request names a `DatabaseMount` (its `id`, `engine`, `credential_ref`, `isolation`, `pool_policy`). The registry maps that mount to a pooled `EnginePool` keyed by `DatabaseMount::effective_pool_key(share_pools)`. If no pool is cached, the adapter's `open_pool(mount)` runs: it asks a `MountResolver` to turn the mount into a concrete DSN, opens the driver pool, and stamps the pool with its isolation flags. The resolver's order is fixed and fail-closed (`EnvMountResolver::resolve_dsn`): an inline DSN wins (parity fast path), then a pluggable `CredentialProvider` (Vault / adapter-registry), then a `db_per_tenant` runtime guard that refuses to borrow the shared map, then the static `DATA_PLANE_MOUNTS` env map, else `CredentialUnavailable`. Once open, every `execute(operation, identity)` runs on that long-lived pool.

**Per-request isolation, not per-pool.** A pool holds connections, never tenant state. Isolation is re-applied on every request from the verified `RequestIdentity`:

- **Postgres / CockroachDB** set RLS GUCs per request inside a short transaction: `apply_rls_context` runs `set_config('app.current_user_id', …, true)`, `set_config('app.current_tenant_id', …, true)`, `set_config('request.jwt.claims', …, true)` with `is_local=true` (transaction-scoped, like `SET LOCAL`). Writes add a second layer of `owner_id` stamping on top of RLS.
- **MySQL / MariaDB / SQLite / MSSQL** have no RLS, so they append an `owner_id = ?` predicate to every read (parenthesising the client filter so a top-level `$or` cannot escape it) and decorate every write with the trusted `owner_id` — stripping any client-supplied `owner_id` / `tenant_id` first, so a forged body can neither read nor write cross-tenant.
- **MongoDB** intersects an `owner_id` + `tenant_id` filter server-side on reads (the client cannot drop the predicate) and overrides those fields on every write document.
- **Redis** namespaces keys (`<owner>:<resource>:<id>`) — the key prefix *is* the isolation.
- **DynamoDB** pins a per-owner partition key (`owner_pk`) plus owner-conditioned writes.
- **HTTP** forwards an `X-Owner-Id` header derived from the identity and leaves authorization to the upstream.

The trusted owner value is consistently `identity.user_id` when present, else `identity.tenant_id` (e.g. the synthetic `api-key:<uuid>` principal).

**Shared pools (B4-pools).** Normally each mount gets its own pool (byte-parity with the pre-B4 behaviour). When `DATA_PLANE_SHARE_POOLS ∈ {1, true, on}` **and** the mount is `shared_rls`, one physical pool serves every tenant. The single source of truth is `pools_shared(mount)` in `lib.rs`; every adapter calls it at `open_pool` to set its `shared_pool` flag. A shared pool skips the defense-in-depth single-tenant `check_tenant` guard (`identity.tenant_id == self.tenant_id`) because the pool is multi-tenant by design — isolation still travels with each request (re-stamped GUCs / `owner_id` predicate / owner key-prefix). `schema_per_tenant` pins a namespace into the pool and `db_per_tenant` / `tenant_owned` use distinct DSNs, so none of those ever share; `DatabaseMount::effective_pool_key` keeps their per-tenant key regardless.

**Capability honesty.** Each adapter exposes a `SUPPORTED_OPS` const that is the single source of truth for both its dispatch gate (unsupported ops error with `NotImplemented`, never silently degrade) and its `EngineCapabilities` descriptor. `capability_honesty.rs` is a test battery asserting `descriptor.supports_op(kind) == SUPPORTED_OPS.contains(kind)` for every engine and op, so the platform can never advertise a capability it doesn't deliver. The server additionally runs a boot-time honesty check over the live adapter list.

**Feature-gated engines.** Every driver is behind a cargo feature (`postgres`, `mongodb`, `mysql`, `redis`, `sqlite`, `mssql`, `http`, plus the optional `dynamodb`). The default build is all of them except `dynamodb`; a lean build (e.g. the nano edition) compiles only what it mounts (`--no-default-features --features sqlite`). Leaving `dynamodb` out of the default keeps the default build from ever fetching or compiling the AWS SDK — byte-parity.

## File-by-file

### `lib.rs`
**Purpose.** Module wiring, feature gates, public re-exports, and the shared-pool helper.
**Key items.** Re-exports each adapter (`PostgresEngineAdapter`, `PgDialect`, `MysqlEngineAdapter`, `MongoEngineAdapter`, `MssqlEngineAdapter`, `RedisEngineAdapter`, `SqliteEngineAdapter`, `HttpEngineAdapter` + `guard_and_resolve`, and under feature `dynamodb`, `DynamoEngineAdapter`), plus `DefaultPoolRegistry`, the resolver (`EnvMountResolver`, `MountResolver`), and the credential layer (`CredentialProvider`, `VaultProvider`, `AdapterRegistryProvider`, `ProviderRegistry`, `ProviderConfig`). The `pub mod service_auth` is public. `capability_honesty` only compiles under `cfg(test)` with every engine feature on.
**The helper.** `pub(crate) fn pools_shared(mount: &DatabaseMount) -> bool` — `true` iff `mount.isolation() == Isolation::SharedRls` AND `DATA_PLANE_SHARE_POOLS ∈ {1, true, on}`. Every adapter calls this at `open_pool`, so the env parse + isolation predicate live in one place.

### `registry.rs`
**Purpose.** `DefaultPoolRegistry`, the `PoolRegistry` impl: maps a mount to a cached `EnginePool`, capped, LRU-evicted, idle-reaped, with rotation drain.
**Key types / functions.** `DefaultPoolRegistry::new` / `with_max_pools` / `with_config(adapters, max_pools, share_pools)`; `pool_key_for(mount)` (the single key-derivation point, → `mount.effective_pool_key(self.share_pools)`); `scale_counters()` → `(created, evicted, reaped, open)` for `/metrics`. `DEFAULT_MAX_POOLS = 256` (overridable via `DATA_PLANE_MAX_POOLS`). Trait methods: `get_or_create`, `release_idle`, `pin_tx` / `unpin_tx`, `close_mount`, `drain_pool_key`, `stats`. A private `SharedPool(Arc<dyn EnginePool>)` wrapper lets the registry hand out owned `Box<dyn EnginePool>` handles that all share one underlying pool, delegating the full `EnginePool` surface (including `execute_raw`, `apply_migration`, `describe_schema`, `apply_schema_ddl`).
**Lifecycle notes.** Eviction is LRU with a hard floor: a pool with an in-flight transaction (`tx_pins > 0`) or the just-created/touched pool (`protect`) is never evicted or reaped — correctness wins over the soft cap. `drain_pool_key` is half of a rotation (the other half is `EnvMountResolver::evict_cached`); it closes the pool for exactly one `pool_key` (which embeds `credential_ref.version`) so the next request rebuilds it with a freshly-resolved credential, leaving a pinned pool for the reaper.
**How it connects.** Built by the server from the adapter list; `get_or_create` calls the matching adapter's `open_pool`.

### `resolver.rs`
**Purpose.** Turn a `DatabaseMount` into a concrete DSN.
**Key types / functions.** `trait MountResolver { async fn resolve_dsn }`; `EnvMountResolver` with constructors `from_json`, `from_env`, `from_config(mounts_json, provider_cfg, cache_ttl_ms)`, and the test seam `with_providers`. A private `CredentialCache` (default OFF, armed by `DATA_PLANE_CREDENTIAL_CACHE_TTL_MS`) wraps provider calls only, keyed by `pool_key`. `evict_cached(pool_key)` is the rotation cache-eviction hook.
**Resolution order (fail-closed, in `resolve_dsn`).** (1) inline `mount.inline_dsn` wins — the parity-critical TS-proxy fast path, so providers never run when inline is present; (2) a registered `CredentialProvider` keyed by `credential_ref.provider` (cache-wrapped); (3) a `db_per_tenant` runtime guard — a `db_per_tenant` mount with no inline DSN and no provider fails closed rather than borrow the shared map (a hard return, not a `debug_assert`); (4) the static `DATA_PLANE_MOUNTS` env map (only reachable for `shared_rls` / `schema_per_tenant`); (5) `CredentialUnavailable`. There is no shared-DSN fallback for any strategy, and an unknown provider name lands at (5).
**How it connects.** Held as `Arc<dyn MountResolver>` by every adapter; called inside `open_pool`.

### `credential.rs`
**Purpose.** Pluggable credential providers (gap G8): resolve a `credential_ref` to a real DSN at runtime without baking secrets into the binary.
**Key types / functions.** `trait CredentialProvider { name(); async resolve_dsn() }`; `ProviderRegistry` (O(1) `HashMap` name → provider; `with`, `from_config`, `from_env`, `get`); `ProviderConfig` (server-owned config, the single env reader); `AdapterRegistryProvider` (name `"adapter-registry"` — `GET /databases/{id}/connect` on the Go control plane, scoped by `X-Baas-Tenant-Id`, authed by service token / HMAC); `VaultProvider` (name `"vault"` — KV v2 read of `…/v1/secret/data/{prefix}/{reference}?version={n}`). `validate_ref_segment` is a fail-closed gate that rejects path-traversal / separators / control chars in `reference` and `version` before they reach an upstream URL (cross-secret confusion guard). Both `ProviderConfig` and `ProviderRegistry` have redacting `Debug` impls (`<redacted>` for set tokens, `""` for empty) — secrets and DSNs never appear in errors or logs; failures return `CredentialProviderFailed { provider, mount_id }` carrying no secret-adjacent input.
**How it connects.** Built by the server from `ProviderConfig`; consulted by `EnvMountResolver` at resolution step (2). Providers are disabled by default (empty registry), so the parity baseline is unchanged.

### `service_auth.rs`
**Purpose.** v1 HMAC service-to-service auth (audit O1) — the Rust caller half of the Go `shared.VerifyServiceRequest`. Under `SERVICE_TOKEN_MODE=hmac` the shared token never transits the wire.
**Key items.** `hmac_mode()`; `compute_service_auth(token, method, path, body)` / `compute_service_auth_at(…, ts)` produce `v1.<ts>.<hex hmac-sha256(token, "<ts>\n<METHOD>\n<PATH>\n<sha256hex(body)>")>`, binding time/method/path/body so a replay against another endpoint or payload fails; `verify_service_auth_at` does a constant-time dual-key verify against the current token and an optional rotation-window `prev_service_token()` (`INTERNAL_SERVICE_TOKEN_PREV`). Golden vectors are shared with the Go and TS implementations so all three sign byte-identically.
**How it connects.** Used by `AdapterRegistryProvider::resolve_dsn` to authenticate the internal `/connect` call (HMAC `X-Service-Auth` header in hmac mode, plain `X-Service-Token` otherwise).

### `ident.rs`
**Purpose.** SQL identifier validation / quoting shared across adapters (injection guard).
**Key items.** `quote_ident(raw)` — double-quoted, accepts an optional single `schema.table` qualifier, each segment `[A-Za-z0-9_]` ≤ 63 chars, else `InvalidIdentifier`. `quote_mysql_ident(raw)` — the backtick variant for MySQL. (Note: `sqlite.rs` has its own module-local `quote_ident`, and `mssql.rs` uses bracket quoting in its own `quote_ident` — see those sections.)

### `tls.rs`
**Purpose.** Shared TLS-posture guards for adapters whose drivers verify by default but can be weakened by a DSN parameter.
**Key items.** `max_security()` (`SECURITY_MODE=max`); `reject_insecure_tls(dsn, max_security, bad_params)` — under max, refuse a DSN carrying a known cert-bypass parameter (`tlsInsecure=true`, `tlsAllowInvalidCertificates=true`, redis `#insecure`, …) rather than silently downgrade. Pure (mode is a parameter) for race-free unit testing. Used by the Mongo and Redis adapters; Postgres and MSSQL have their own dedicated posture (`postgres::effective_tls_mode`, `mssql::apply_mssql_tls`).

### `capability_honesty.rs`
**Purpose.** The test battery enforcing "descriptors must not lie."
**Key items.** `dispatch_reality(engine)` returns the very `SUPPORTED_OPS` const each adapter's dispatch gate uses (not a hand copy); `descriptor(engine)` returns the `EngineCapabilities` the adapter advertises. `descriptor_advertises_exactly_what_dispatch_implements` asserts they agree for every op of every engine; further tests pin the `batch` flag (true for pg/mysql/mongo/redis, false for http) and the `transactions` flag (true for pg/mysql, false for mongo/redis/http) against the actual `begin()` implementations. A `cfg(feature = "dynamodb")` test extends the same check to DynamoDB (`transactions` + `native_idempotency` true, `aggregate` false).

### Engine adapters

All eight follow the same shape: a `…EngineAdapter` (`EngineAdapter` impl, holds `Arc<dyn MountResolver>`), a `…Pool` (`EnginePool` impl, one per mount, carries `mount_id` / `tenant_id` / `shared_pool`), a `pub(crate) const SUPPORTED_OPS`, and a `check_tenant`-style single-owner guard that is skipped when `shared_pool` is true. `health_check` is a stub returning `status: "unknown"` on every engine.

#### `postgres.rs` — PostgreSQL / CockroachDB
- **Driver / pool.** `tokio-postgres` + `deadpool-postgres` (`deadpool_postgres::Pool`), TLS via `tokio-postgres-rustls` (`MakeRustlsConnect`) over `rustls` with `webpki-roots`. Structs: `PostgresEngineAdapter` (`new` / `with_dialect`), `PostgresPool`, `PgTxHandle` (pins a `deadpool_postgres::Object` for the tx's life). `PgDialect::{Postgres, Cockroach}` rides the same adapter over pgwire — `engine()` → `postgresql` vs `cockroachdb`, descriptor `postgresql()` vs `cockroachdb()` (Cockroach: `stream: false`, serializable-only; emits plain `BEGIN`).
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Aggregate, Batch`. Transactions + atomic batch + savepoints (`begin()` returns a real `PgTxHandle`; 2PC `prepare()` is `NotImplemented`).
- **Isolation & capability notes.** RLS GUCs re-stamped per request inside a short transaction (`apply_rls_context`, `is_local=true`); `request.jwt.claims` is built with `serde_json::json!` so an injected `sub` can't be smuggled. Reads rely on the GUC context; writes add an `owner_id` second layer — `run_insert` strips and re-injects the trusted principal, `run_update`/`run_delete` append `AND "owner_id" = $n` and refuse an unconstrained filter, `run_upsert` forces `owner_id` into the `ON CONFLICT (owner_id, …)` target and keeps it immutable. `schema_per_tenant` resolves the schema once at `open_pool` and runs `SET LOCAL search_path` per tx. `tenant_owned` (`owner_scoped()==false`) skips all owner machinery. `shared_pool` bypasses `check_tenant`. TLS: `effective_tls_mode` reads `sslmode` from the DSN; `SECURITY_MODE=max` upgrades a bare `Require` to chain+hostname `Verify` (optional `DATA_PLANE_TLS_CA_FILE`).
- **Quirks.** Constraint errors (SQLSTATE class 23 / class 22) → `Conflict` (409, avoids outbox retry loops); `42P10` (no matching `ON CONFLICT` constraint) → `InvalidRequest` (400) with a message about the `(owner_id, key…)` composite UNIQUE. A custom `JsonParam` `ToSql` chooses the binary encoding from the *target column type* (int widths, `NUMERIC` via a hand-rolled base-10000 encoder, UUID/timestamp/enum). `apply_migration` / `execute_raw` are admin-scoped and bypass RLS (route-gated); migrations are idempotent via a `_baas_migrations` marker table. `apply_schema_ddl` lowers via pure `build_pg_ddl` builders and auto-synthesizes an `owner_id text` column on owner-scoped `CreateTable`. Optional pooler knobs: `DATA_PLANE_POOLER_URL`, `DATA_PLANE_STATEMENT_CACHE=off`. (No `pg_graphql` here — GraphQL is a separate passthrough route.)

#### `mysql.rs` — MySQL / MariaDB
- **Driver / pool.** `mysql_async` with its built-in `Pool`. Structs: `MysqlEngineAdapter` (`new` / `with_engine_name`), `MysqlPool`, `MysqlTxHandle` (pins a `mysql_async::Conn`). One adapter serves both engines parameterized by `engine_name` — descriptor `mysql()` vs `mariadb()`, no dialect enum.
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Aggregate, Batch`. Per-request auto-commit transaction; `begin()` returns a `MysqlTxHandle`; atomic batch; `prepare()` (XA 2PC) `NotImplemented`.
- **Isolation & capability notes.** No RLS. `RESERVED_COLUMNS = ["owner_id"]`. `build_owner_filter` parenthesises the client filter then `AND`s `` `owner_id` = ? `` (so a top-level `$or` can't escape — proven by `owner_filter_lowers_operators_not_just_equality`); `build_owned_columns` strips client `owner_id` and re-injects the trusted one. Identifiers via `crate::ident::quote_mysql_ident` (backticks). `schema_per_tenant` issues `USE \`schema\`` on every checkout. `check_tenant` skipped when `shared_pool`; `open_pool` fails closed (`NotImplemented`) for non-owner-scoped isolation (Postgres-only `tenant_owned`).
- **Quirks.** Upsert is `ON DUPLICATE KEY UPDATE` guarded with `IF(\`owner_id\` = VALUES(\`owner_id\`), …)` so a foreign-owner collision is a no-op, not a hijack. `SUM`/`DECIMAL` arrive as `Bytes` → surface as JSON strings. Errors: duplicate-entry / FK / truncation / range (1062/1452/1264/1265/1292/1366) → `Conflict`; DDL path also maps already-exists → `Conflict`, unknown-column → `InvalidRequest`. Update/delete refuse an unconstrained filter. `$ilike` lowers to `LOWER(col) LIKE LOWER(?)`. **No TLS hardening** — TLS is purely DSN-driven through `Opts::from_url`; the `SECURITY_MODE=max` guard does *not* apply here (an asymmetry vs Postgres/Mongo/Redis). `apply_migration` uses a `_baas_migrations` marker (DDL implicit-commits, so not all-or-nothing); `CreateTable` auto-appends `` `owner_id` VARCHAR(64) `` (sized for `api-key:<uuid>` principals).

#### `mongo.rs` — MongoDB
- **Driver / pool.** The official `mongodb` 2.8 driver + `bson`; the driver owns a connection pool per `Client`. Structs: `MongoEngineAdapter` (`new`), `MongoPool` (wraps `mongodb::Client`).
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Aggregate, Batch`. `begin()` returns `NotImplemented` (multi-statement tx deferred — single-document writes stay atomic); batch is **ordered, non-atomic** (`BatchSummary { atomic: false }`).
- **Isolation & capability notes.** `RESERVED_FIELDS = ["_id", "owner_id", "tenant_id"]` (stripped from writes); `FILTER_TRUST_FIELDS = ["owner_id", "tenant_id"]` (stripped from filters — `_id` deliberately kept so by-PK ops still select one row). `build_tenant_filter` intersects the client filter with server-inserted `owner_id` + `tenant_id` (the client cannot drop the predicate; rejects unsafe `$` operators); `build_owned_doc` overrides the same fields on every write. The `tenant_id` stamp is sourced from `identity.tenant_id`, not the pool field — so it travels with the request under shared pools. TLS: `reject_insecure_tls(dsn, max_security(), &["tlsinsecure=true", "tlsallowinvalidcertificates=true", "tlsallowinvalidhostnames=true"])`. `check_tenant` skipped when `shared_pool`; non-owner-scoped isolation fails closed.
- **Quirks.** `normalize_doc` maps Mongo `_id` → `id` without clobbering a client logical `id`; `coerce_id_filter` dual-matches a 24-hex string as ObjectId OR literal string. Upsert requires a scalar `id` (rejects object `_id` to block `{$gt:""}` injection). Aggregate lowers to `$match → $group → $project` reusing `build_tenant_filter` (distinct aggregates rejected). `SAFE_MONGO_OPERATORS` is a default-deny allowlist blocking `$where`/`$expr`/`$function`/server-side JS. Errors: duplicate-key (E11000) / validation → `Conflict`. `describe_schema` uses a `$jsonSchema` validator if present, else infers from a 200-doc `$sample`; DDL maps onto the validator.

#### `sqlite.rs` — embedded SQLite
- **Driver / pool.** `rusqlite` (bundled) driven through `deadpool-sqlite` (`obj.interact(…)` on a blocking thread). Structs: `SqliteEngineAdapter` (`new`), `SqlitePool` (deadpool readers + a dedicated writer channel). File-per-mount; `sqlite_path` parses the DSN (`sqlite://…`, empty / `:memory:` → in-memory).
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Aggregate, Batch`. `begin()` returns `NotImplemented` (`transactions: false`); batch is **atomic** (a poison item rolls the whole batch back).
- **Isolation & capability notes.** No RLS. `RESERVED_COLUMNS = ["owner_id", "tenant_id"]`. `build_owner_filter` appends `"owner_id" = ?` on reads/update/delete (when owner-scoped); `build_owned_columns` re-injects the trusted `owner_id` on writes; update/delete also call `guard_constraining_filter` against full-table mutation. `owner_scoped` from `mount.isolation().owner_scoped()`. Local module `quote_ident` (double-quoted, ≤128, rejects `"`/NUL/control). `check_tenant` skipped when `shared_pool`.
- **Quirks.** Upsert uses real `ON CONFLICT (cols) DO UPDATE/DO NOTHING`; a foreign-owner `id` collision hits the PK (unhandled conflict target) and errors rather than overwriting. Errors: unique / constraint / not-null / FK → `Conflict`; DDL already-exists / no-such-table/column → `InvalidRequest`. **Concurrency is the notable part:** WAL + one OS writer thread per mount (`sqlite-writer-<id>`, `writer_loop`) group-commits up to `GROUP_MAX = 128` jobs in one `BEGIN IMMEDIATE`, each wrapped in a `SAVEPOINT` for per-job atomicity; reads stay parallel on the pool. `AlterColumnType` is honestly rejected (SQLite has no `ALTER COLUMN`); `CreateTable` auto-appends `owner_id TEXT`; enums lower to `TEXT` + `CHECK (… IN (…))`. Arrays/objects stored as JSON text, blobs read back lossy as `"blob:N bytes"`.

#### `dynamodb.rs` — DynamoDB / DynamoDB-Local / ScyllaDB Alternator (optional `dynamodb` feature, OFF by default)
- **Driver / pool.** `aws-sdk-dynamodb` + `aws-config` (`BehaviorVersion::latest()`); one endpoint-agnostic adapter selected by DSN (`endpoint=` → Local/Scylla, else real AWS; optional `access_key`/`secret_key`). Structs: `DynamoEngineAdapter` (`new`), `DynamoPool` (wraps the `Arc`-cheap `Client`), `DynamoTxHandle`.
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Batch` — **`Aggregate` deliberately absent** (no server-side GROUP BY/SUM; OLAP is the export bridge). Descriptor `dynamodb()`: `transactions: true`, `native_idempotency: true`, `aggregate: false`, `max_batch_size: 25`.
- **Isolation & capability notes.** No RLS — a composite key carries isolation: `PK = "owner_pk"`, `SK = "id"`. `build_owner_pk` yields `<namespace>:<owner>` (or bare `<owner>`); `!owner_scoped` → fixed sentinel `"_tenant_owned"`. Writes stamp `owner_pk` + a redundant `owner` attr; reads are single-partition (`Get` keys on `(PK, SK)`, `List` is a `Query` with `#pk = :owner`). Writes are owner-conditioned: insert `attribute_not_exists(#pk)`, update `attribute_exists(#pk) AND #owner = :owner`, delete `attribute_exists(#pk)`. The tenant-match guard is skipped when `shared_pool`.
- **Quirks.** CRUD → PutItem/GetItem/Query/UpdateItem(full-replace via PutItem)/DeleteItem; upsert is unconditional last-writer-wins. `begin()` is **buffer-then-commit**: each tx op pushes a `TransactWriteItem`, `commit()` issues one `transact_write_items` with a `ClientRequestToken` (native idempotency); reads inside a tx are rejected. Batch is ordered, non-atomic, capped at 25. Errors: `ConditionalCheckFailedException` (insert collision) / `TransactionCanceledException` → `Conflict` (409, detected by debug-text substring); update/delete on a missing/foreign item map to a 0-row result, not an error. Nested JSON is an MVP shim (stored as a JSON-string attribute).

#### `mssql.rs` — Microsoft SQL Server
- **Driver / pool.** `tiberius` (TDS, fully async, no `spawn_blocking`) pooled via `bb8` + `bb8-tiberius` (`Pool<ConnectionManager>`). Structs: `MssqlEngineAdapter` (`new`), `MssqlPool`. DSN `mssql://` / `sqlserver://` (default port 1433, db `master`, `AuthMethod::sql_server`).
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert, Aggregate, Batch`. `begin()` returns `NotImplemented` (`transactions: false`); batch is atomic (`BEGIN TRAN` on one pooled connection); upsert via T-SQL `MERGE`.
- **Isolation & capability notes.** No RLS (SQL Server RLS is not assumed). `RESERVED_COLUMNS = ["owner_id", "tenant_id"]`; `build_owner_filter` appends `[owner_id] = @Pn`, `build_owned_columns` stamps `owner_id` on writes; client `owner_id`/`tenant_id` stripped from filters and data. Identifiers via bracket-quoting `quote_ident` (`[name]`, `]` → `]]`, ≤128). `check_tenant` skipped when `shared_pool`. TLS via `apply_mssql_tls(config)`: closes the old unconditional `trust_cert()` hole — under `SECURITY_MODE=max` the `DATA_PLANE_TLS_INSECURE` escape is ignored; `DATA_PLANE_TLS_CA_FILE` pins a custom CA; otherwise native-root verification.
- **Quirks.** Parameters are `@P1`-style (1-indexed), bound via a `P` enum implementing `tiberius::ToSql`. `MERGE` arbitrates on `(owner_id, sorted filter keys)` — a foreign-owner id collision attempts INSERT → PK violation rather than overwrite. Pagination needs `ORDER BY`, so `build_list` synthesizes `ORDER BY (SELECT NULL)` when no sort is given. Errors: unique / duplicate-key / PK / FK / null / constraint → `Conflict`. Full-table mutation guard via `guard_constraining_filter`. Introspection only (`describe_schema` over `INFORMATION_SCHEMA`, no PK populated); no DDL execution path; `execute_raw` allows raw SQL with bound params.

#### `redis.rs` — Redis (KV / Hash)
- **Driver / pool.** The `redis` crate with an auto-reconnecting `ConnectionManager` (Arc-cheap, one per mount). Structs: `RedisEngineAdapter` (`new`), `RedisPool`. Each record is a Redis Hash.
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert` (no `Aggregate`, no `Batch` in the const — but `run_batch` exists, ordered + non-atomic, `BatchSummary { atomic: false }`). `begin()` → `NotImplemented` (MULTI/EXEC not exposed); aggregate → `NotImplemented`.
- **Isolation & capability notes.** No server-side predicate — **the key namespace is the isolation.** `build_key_prefix` yields `<owner>:<resource>` (or `<namespace>:<owner>:<resource>` for `schema_per_tenant`); the full key is `<prefix>:<id>`. Reads and writes derive the key from the same identity-derived prefix, so a forged `id` can't reach another tenant's keyspace. TLS: `reject_insecure_tls(dsn, max_security(), &["#insecure"])` rejects `rediss://…#insecure` under max. Single-owner guard skipped when `shared_pool`.
- **Quirks.** `List` is a `SCAN MATCH <prefix>:*` (non-blocking, unlike `KEYS`) then `HGETALL` per key, sorted and sliced client-side (no cursor; `next_cursor` always `None`). `Get`→`HGETALL`, `Insert`→`EXISTS` guard then `HSET`, `Update`→`EXISTS` guard (missing → 0 rows, no error), `Delete`→`DEL`, `Upsert`→blind `HSET`. Per-field values stored via `value_to_hash_string` (strings as-is, else `serde_json::to_string`) and parsed back best-effort on read. Ids from `filter.id`/`data.id`; generated when absent (`generate_id`, not crypto). Resource/id charset validated to protect the `:` envelope.

#### `http.rs` — HTTP passthrough
- **Driver / pool.** `reqwest` (its `Client` owns its own connection pool). Structs: `HttpEngineAdapter` (`new`), `HttpPool`. The mount's connection string is JSON `{baseUrl, headers?, routes?}` or a bare http(s) URL.
- **Ops.** `SUPPORTED_OPS = List, Get, Insert, Update, Delete, Upsert` (CRUD → GET/GET/POST/PATCH/DELETE/PUT). `begin()` and batch/aggregate → `NotImplemented` (a remote REST passthrough cannot give those semantics — honesty over uniformity; descriptor `batch: false`).
- **Isolation & capability notes.** An HTTP mount has no schema/db/keyspace, so it applies **no per-request scoping** under any strategy — instead it forwards an `X-Owner-Id` header (`identity.user_id` or `tenant_id`) so the upstream applies its own authz. Documented as an explicit no-op. The single-tenant guard is skipped when `shared_pool`. **SSRF guard:** `pub async fn guard_and_resolve(base_url)` (re-exported) resolves and validates the host against `is_blocked_ip` (loopback, RFC-1918, link-local incl. `169.254.169.254`, CGNAT, IPv6 ULA/link-local, IPv4-mapped forms) and blocks names like `localhost` / `*.internal` / `metadata`; it pins the client to the validated public IP(s) so a later DNS rebind can't redirect inward. `DATA_PLANE_HTTP_ALLOW_INTERNAL=1` skips it for trusted dev mocks.
- **Quirks.** Custom `routes` override default paths per op; filter/sort/limit/offset are URL-encoded into the query string; the idempotency key is forwarded as an `idempotency-key` header. Responses are shaped to `DataResult` (array → rows; `{data:[…]}` → rows; object → 1 row). Upstream 4xx/5xx → `Backend`.

## The adapter pipeline

Trace a `POST /v1/query` for one mount:

1. **Resolve.** The server constructs a `DatabaseMount` (engine, `credential_ref`, `isolation`, `inline_dsn?`). The adapter's `open_pool` calls `MountResolver::resolve_dsn(&mount)` — `EnvMountResolver` runs its fixed order: inline DSN → `ProviderRegistry::get(provider)` (cache-wrapped) → `db_per_tenant` fail-closed guard → `DATA_PLANE_MOUNTS` map → `CredentialUnavailable`.
2. **Register.** The server calls `DefaultPoolRegistry::get_or_create(mount)`. It keys on `mount.effective_pool_key(self.share_pools)`; a cache hit returns a `SharedPool` clone, a miss calls the matching adapter's `open_pool`, inserts the entry, and runs LRU `collect_evictions` (protecting the new pool and any `tx_pins > 0` pool).
3. **Credential.** Inside `open_pool` the resolved DSN becomes the driver pool; `pools_shared(&mount)` sets the pool's `shared_pool` flag; the pool caches its schema/namespace if the strategy needs one.
4. **Checkout + execute.** The server calls `EnginePool::execute(operation, identity)`. The pool runs its `check_tenant` guard (skipped if `shared_pool`), applies isolation for the request (RLS GUCs / `owner_id` predicate + write decoration / key namespace / `owner_pk` / `X-Owner-Id`), dispatches the op against `SUPPORTED_OPS` (anything else → `NotImplemented`), and returns a `DataResult`.
5. **Transactions.** For `begin(request)`, the pool returns a `TxHandle` (where supported) and the server calls `pin_tx(pool_key)` to protect the pool from eviction/reaping until the tx finishes (`unpin_tx`).
6. **Rotation.** `AppState::rotate` (over `POST /v1/admin/rotate`) calls both `DefaultPoolRegistry::drain_pool_key` and `EnvMountResolver::evict_cached` so the next request rebuilds the pool with a freshly-resolved credential.

## Feature flags

From `Cargo.toml`. `default = ["postgres", "mongodb", "mysql", "redis", "sqlite", "mssql", "http"]` — `dynamodb` is intentionally excluded so the default build never fetches or compiles the AWS SDK.

| Feature | Engine(s) | Pulls in |
|---------|-----------|----------|
| `postgres` | PostgreSQL / CockroachDB | `tokio-postgres`, `deadpool-postgres`, `tokio-postgres-rustls`, `rustls`, `webpki-roots`, `rustls-pemfile`, `bytes` |
| `mongodb` | MongoDB | `mongodb`, `bson` |
| `mysql` | MySQL / MariaDB | `mysql_async` |
| `redis` | Redis | `redis` |
| `sqlite` | embedded SQLite | `rusqlite`, `deadpool-sqlite` |
| `mssql` | Microsoft SQL Server | `tiberius`, `bb8`, `bb8-tiberius`, `tokio-util` |
| `http` | HTTP passthrough | `percent-encoding` |
| `dynamodb` (off by default) | DynamoDB / DynamoDB-Local / ScyllaDB Alternator | `aws-sdk-dynamodb`, `aws-config` |

Non-optional deps used by every build include `data-plane-core`, `async-trait`, `chrono`, `uuid`, `futures`, `reqwest`, `serde`/`serde_json`, `tokio`, `tracing`, and `hmac`/`sha2` (service-auth). A lean build picks a subset, e.g. `--no-default-features --features sqlite`; the full default build compiles every engine and the entire conformance / capability-honesty battery.
